第二章 缓冲区溢出攻击

什么是shellcode

Shellcode是一段特定的机器代码，通常用于利用软件漏洞或进行恶意操作。它是一系列计算机指令的二进制表示形式，设计用于在计算机系统上执行特定任务。

Shellcode通常用于利用软件的安全漏洞，如缓冲区溢出、注入攻击等。攻击者通过将Shellcode注入到受攻击的程序中，可以在目标系统上执行任意指令，从而实现攻击的目的。这可能包括获取系统权限、执行恶意代码、绕过安全措施等。

缓冲区溢出攻击

  • 基本原理：像缓冲区写入超出其容量的数据，覆盖相邻内存区域，特别是return address部分，并利用返回地址来跳转执行恶意代码shellcode
  • 根本原因：C/CPP没对数组引用进行边界检查

利用strcpy栈溢出漏洞得到shell的过程

大致流程

  1. 使用一个字符数组存放shellcode[既要包含shellcode，还要存放return address确保返回地址成功跳转，最难的是定位溢出的位置]
  2. 利用strcpy函数，将shellcode放入程序的堆栈中
  3. 我们需要制造数组越界，所以拷贝的目标数组大小小于shellcode长度，确保shellcode的起始地址覆盖了程序原本的返回地址return address
  4. 在程序返回时，由于返回地址被修改为shellcode的起始地址，因此会执行shellcode，从而获得shell

shellcode的设计

实行类似exec族函数获得被攻击主机的shell，以获得root权限

只有shellcode的机器代码才能植入到缓冲区，所以要先编译成汇编代码之后再得到适合主机的机器代码

exploit攻击程序的设计

char shellcode[] = "......" //45 chars的攻击机器码
char large_string[128]; //128字节的大字符串

int main(){
char buffer[96]; //缓冲区大小96字节
int i;
long * long_ptr = (long *) large_string; //指针指向大字符串的首地址
for(i = 0 ; i < 32 ; i ++)
    *(long_ptr + i) = (int) buffer; //将buffer的首地址填满long_string
for(i = 0 ; i < strlen(shellcode) ; i ++)
    large_string[i] = shellcode[i]; //shellcode内容复制到large_string
strcpy(buffer,large_string); //缓冲区溢出攻击的重点
return 0;
}

第四章 高层协议安全

HTTP与HTTPS

HTTPS = HTTP + 加密 + 认证 +完整性保护

SSL/TLS安全套接字

SSL（Secure Socket Layer）安全套接字提供两个基本的安全服务：鉴别和保密

SSL采用的加密技术为：对称加密技术

分为三个协议

  • 握手协议（Handshake Protocol）
  • 记录协议（Record Protocol）
  • 警报协议（Alert Protocol）

握手协议

  1. 客户端发起HTTPS请求（发送支持的加密协议+随机生成的不重数1）
  2. 服务器端配置（选择一个加密协议+生成不重数2）
  3. 服务器返回（包含了经第三方CA认证的公钥（crt public）+不重数2）
  4. 客户端验证证书，通过后生成预主密钥（PMK）
  5. 用服务器的公钥crt public加密PMK后发送给服务器
  6. 服务器用crt private解密后得到PMK
  7. 双方通过将数1+数2+PMK => MK（会话的主密钥）

记录协议

记录协议提供了保密性和完整性

  • 保密性：使用握手协议定义的主密钥（对称）进行消息的加密传递
  • 完整性：握手协议定义了MAC（消息验证码）用于保证信息的完整性

警报协议

分为两个字节：

  • 第一个字节（1:警报；2:致命错误）
  • 第二个字节（记录实际的错误类型）

二阶段SQL注入与普通注入的区别

普通注入是在单个请求中直接注入恶意的SQL代码，而二阶段SQL注入是在第一次注入时留下恶意数据，并在后续的操作中利用这些数据进行攻击。

[个人理解：可以类比为XSS中的反射型XSS和存储型XSS]

普通注入是指直接将恶意的SQL代码注入到应用程序的用户输入字段中，然后通过执行注入的代码来绕过应用程序的安全机制，对数据库进行未经授权的访问或操作。这种注入方式通常发生在单个请求中，攻击者能够直接控制注入的数据内容，从而影响整个查询或命令的执行。

而二阶段SQL注入则是在应用程序中，用户的输入数据被存储在数据库中，并在后续的查询或操作中被使用。攻击者在第一次注入时，可能并不会直接获得所需的结果，但他们将恶意的数据留在数据库中，并等待应用程序在后续的操作中使用这些数据。当应用程序执行后续操作时，恶意数据被再次使用，并导致注入攻击的成功。这种方式需要攻击者对应用程序的工作流程和数据库的结构有一定的了解，以便在后续的操作中利用之前注入的数据。

盲SQL 注入

与普通SQL注入的区别

盲SQL注入与普通的SQL注入类似，都是利用应用程序没有对用户输入的数据进行足够的过滤和验证，导致攻击者可以向数据库发送自己构造的恶意SQL语句。

但是不同之处在于，盲SQL注入攻击者无法直接获取查询结果，而只能通过推断和试错的方式来获取信息。盲SQL注入可以分为基于布尔逻辑的盲注和基于时间延迟的盲注两种类型。

基于布尔逻辑的盲注

攻击者通过构造特定的SQL语句，利用应用程序对查询结果的布尔逻辑判断返回不同的响应（是否返回错误信息），从而推断出数据库中的信息。

SELECT * FROM users WHERE username = 'admin' AND password LIKE '%a%'

这个SQL查询语句包含了一个布尔逻辑判断条件，即password字段中是否包含字母a。如果该条件成立，应用程序会返回正常的查询结果，否则应用程序会返回错误信息。

SELECT * FROM users WHERE username = 'admin' AND password LIKE 'a%'

如果应用程序返回了正常的查询结果，说明password字段以字母a开头，否则说明不是。通过这种方式，攻击者可以逐个字母地推断出password字段的值，从而获取管理员账号的密码等敏感信息。

基于时间延迟的盲注

基于时间延迟的盲注是一种盲注攻击的变体，它通过在注入的SQL语句中嵌入条件，并利用应用程序的响应时间差异来推断条件的真假。

攻击者可以构造带有时间延迟的SQL语句，并观察应用程序的响应时间来判断条件是否成立。例如，攻击者可以构造以下SQL语句：

SELECT * FROM users WHERE username='admin' AND IF(SUBSTR(password,1,1)='a' waitfor delay '0:0:5';

如果密码的第一个字母是a，则会有5秒的延迟。

如果应用程序响应时间较长，则说明条件成立，反之则说明条件不成立。通过逐个条件地推断，攻击者可以逐渐获取数据库中的信息，例如用户名、密码等敏感数据。

SQL注入攻击的预防

通过使用参数化查询（也称为预处理）而不是在查询中进行字符串拼接，可以防止大多数SQL注入攻击。以下代码由于用户输入直接被拼接到查询中，因此容易受到SQL注入攻击

String query = "SELECT * FROM products WHERE category = '" + input + "'";
Statement statement = connection.createStatement();
ResultSet resultSet = statement.executeQuery(query);

可以通过简单地重写该代码，防止用户输入干扰查询结构：

Prepared Statement statement = connection.prepareStatement("SELECT * FROM products WHERE category = ?");

statement.setString(1 , input);
//将input赋值给第 1 个参数
ResultSet resultSet = statement.executeGuery();

文件遍历

文件遍历攻击（也称为目录遍历攻击、路径遍历攻击或. ./攻击）是一种常见的Web应用程序攻击，其目的是访问服务器上未经授权的文件或目录。攻击者通过构造包含特殊字符（如../）的URL或输入表单，试图跳出Web应用程序根目录，访问服务器上的其他目录或文件。

一旦攻击成功，攻击者可以访问包含敏感信息的文件，如配置文件、密码文件、数据库文件等，从而对系统进行进一步的攻击或窃取敏感信息。

点击劫持

点击劫持是一种基于界面的攻击，攻击者通过在诱骗网站上点击其他内容来欺骗用户点击一个隐藏网站上的可操作内容。考虑以下示例：一个网络用户访问了一个诱骗网站（可能是通过电子邮件提供的链接），并点击一个按钮以赢取奖品。不知情的用户被攻击者欺骗，实际上按下了一个隐藏按钮，导致在另一个网站上支付了一个账户。这就是点击劫持攻击的一个例子。这种技术依赖于将一个不可见的、可操作的网页（或多个页面）与一个包含按钮或隐藏链接的iframe结合起来。这个iframe会覆盖在用户预期的诱骗网页内容之上。这种攻击与CSRF攻击不同，因为用户需要执行一个动作，比如点击一个按钮，而CSRF攻击则依赖于在用户不知情或无需用户输入的情况下伪造一个完整的请求。

iframe

iframe是点击劫持攻击发起的基础

iframe允许将浏览器窗口分割为不同的部分，每个部分可以显示不同的文档（这些文档可以属于同一个dom域，也可以属于不同的dom域）

同源策略（same-origin policy）规定了一个域名下的DOM对象不能访问来自不同域名的DOM对象。因此对于网页中存在的类似于Facebook like 按钮由于其属于facebook.com，因此不与主网页共享同一个域名，like按钮的cookie也无法被窃取

如果想要骗取点赞，只能使用点击劫持，比如在like按钮上方覆盖一个 get a FREE iPad 按钮。

点击劫持如何去防御？

延迟（Mitigation），重定向（framebusting）和响应头（X-FRAME-OPTION）

  • Mitigation：在按钮出现和可使用之间添加延迟（例如，对Install按钮采取等待10s后点击有效，这样 如果install被伪装成别的按钮后，弹出的10s内被用户点击，不会执行install程序）可以帮助预防点击劫持。
  • framebusting：Framebusting是一种技术，当一个页面被嵌入到一个框架中时，可以停止其正常功能。

framebusting的代码一般由一个条件表达式和纠正动作（跳转）组成，即将顶层页面（z-index值最大的页面）导航至当前页面【把top.location（覆盖在原始页面上的“恶意”frame）重定向回self.frame（“正确”的frame）】

这个脚本会检测页面是否在iframe中运行，如果是，则会通过修改窗口的top位置来使页面跳出当前的框架，并替换嵌入页面，从而防止点击劫持攻击的进行。

Framebusting脚本的工作原理是利用不同域的页面无法修改其顶级窗口的属性，因此通过修改top位置来实现跳出框架。一旦页面跳出了iframe，它将以独立的窗口显示，从而避免了点击劫持攻击。

  • X-FRAME-OPTIONS：X-FRAME-OPTION一共有三个值，分别为DENY（任何页面都不能被嵌入到iframe或frame），SAMEORIGIN（页面只能被本站嵌入到iframe或frame）和ALLOW-FROM URL（页面只能被指定的URL嵌入到iframe或frame）;通过设置X-Frame-Options头部，网站管理员可以告知浏览器如何处理页面的嵌入行为。这样可以有效地防止点击劫持攻击，因为浏览器会遵循相应的策略来限制页面的嵌入。

跨站脚本攻击XSS

XSS反射型攻击

步骤

1 钓鱼

2 访问链接

3 回来一个带恶意脚本的response

4 获取一个cookies 返回去

反射型XSS攻击的步骤通常如下

  1. 攻击者构造带有恶意脚本的URL，这个URL通常会伪装成一个正常的URL，比如通过发送给受害者一个带有恶意脚本的电子邮件，或者在社交媒体平台上发布一个包含恶意脚本的链接等方式来引诱受害者点击。
  2. 受害者点击链接后，浏览器会向服务器发送一个请求，请求会包含攻击者构造的恶意脚本。
  3. 服务器接收到请求后，会将恶意脚本作为响应的一部分返回给浏览器。
  4. 受害者的浏览器解析服务器返回的响应，并执行其中包含的恶意脚本。
  5. 恶意脚本被执行后，可以窃取受害者的敏感信息，如登录凭据、会话令牌等，并将这些信息发送给攻击者。
  6. 攻击者接收到窃取的信息后，可以利用这些信息进行各种恶意行为，如冒充受害者进行欺诈、窃取个人信息等。

需要注意的是，反射型XSS攻击通常只能攻击单个受害者，因此攻击者通常需要通过大量的钓鱼邮件或社交媒体链接来引诱尽可能多的用户点击包含恶意脚本的链接，以增加攻击的成功率。

如何构造恶意脚本的URL

  1. 首先，攻击者需要找到目标网站上可以触发反射型XSS漏洞的页面。这通常需要对目标网站进行漏洞扫描或手动测试。
  2. 然后，攻击者需要构造一个带有恶意脚本的URL。恶意脚本可以是JavaScript代码或其他客户端脚本，它通常被嵌入到URL的查询字符串参数中。例如，攻击者可以构造一个包含以下查询字符串参数的URL：http://example.com/search?query=<script>alert('XSS')</script>

在这个例子中，攻击者在查询字符串参数中嵌入了一个弹窗恶意脚本，当用户访问这个URL时，浏览器会执行这段脚本并显示一个弹窗。

反射型XSS最大的特点：它的攻击方式：与其他类型的XSS攻击（如存储型XSS攻击）不同，反射型XSS攻击不需要攻击者将恶意脚本注入到服务器端的数据库中，而是利用受害者自己的浏览器来执行恶意脚本。

“反射型XSS攻击”之所以被称为“反射型”，是因为攻击的核心机制是将恶意脚本通过受害者的请求“反射”回响应中，然后由浏览器解析和执行这些恶意脚本。

存储型XSS

存储型XSS（Stored XSS）攻击是一种常见的Web应用程序漏洞，攻击者通过将恶意脚本存储到目标网站的数据库或文件系统中，从而在目标网站的页面中注入恶意脚本，达到攻击的目的。

存储型XSS攻击与反射型XSS攻击不同，存储型XSS攻击的恶意脚本并不是反射到响应中，而是存储在目标网站的数据库或文件系统中。攻击者通常通过提交恶意数据来触发存储型XSS漏洞，例如通过提交包含恶意脚本的评论、留言、表单数据等。一旦目标网站将恶意数据存储到数据库或文件系统中，并在后续的页面中展示这些数据，恶意脚本就会被执行，从而实现攻击者的目的。

简单的说，反射型是浏览器在提交请求时附带了恶意脚本，当服务器返回时，浏览器需要去处理服务器返回的，这叫做将恶意脚本反射到响应中；存储型XSS则是存储在目标网站中的，本地浏览器在展示目标网站页面的时候（评论，留言等）自然的会去执行恶意脚本代码

跨站请求伪造CSRF

CSRF 攻击的工作原理如下：

  1. 用户登录：用户在一个网站（网站A）上进行登录，并获取了有效的会话凭证（例如，通过使用 cookie）。
  2. 攻击网站准备：攻击者在另一个网站（网站B）上准备了一个恶意页面或陷阱。
  3. 恶意页面包含请求：恶意页面中包含了针对网站A的恶意请求，例如转账、更改密码等操作。该请求通常是以隐藏的形式嵌入在页面的内容中。
  4. 用户访问恶意页面：用户在未注销网站A的情况下，访问了网站B上的恶意页面。
  5. 自动提交请求：恶意页面中的恶意请求被自动触发，由于用户已经在网站A上登录并具有有效的会话凭证，所以该请求会被发送到网站A。
  6. 恶意请求执行：网站A接收到恶意请求并处理它，由于请求看起来是由受信任的用户发送的，因此网站A可能会执行该请求，并完成攻击者想要的恶意操作。

能成功的原因是：在同一个浏览器下，无论是哪个页面或网站向a网站发送请求，请求头都会带上a.com域名下的cookie

CSRF防御技术

CSRF token

CSRF令牌（CSRF token）是一种常见的用于防御CSRF攻击的技术。CSRF令牌是一个随机生成的令牌（token），通常以密钥对的形式存在于用户的会话中。

当用户登录到应用程序时，服务器会生成一个唯一的CSRF令牌并将其与用户的会话关联起来。在用户执行重要操作（如提交表单、更改密码等）时，应用程序会将CSRF令牌嵌入到请求中，通常以隐藏字段的形式添加到表单中或作为请求参数。

当服务器接收到请求时，它会验证请求中的CSRF令牌是否与用户会话中的令牌匹配。如果匹配成功，则说明请求是合法的，服务器会处理该请求。如果令牌不匹配或缺失，则服务器会拒绝该请求，因为它可能是由攻击者发起的CSRF攻击。

CSRF令牌是一种有效的防御技术，因为攻击者很难获取到用户的有效令牌，并且每个用户的令牌都是唯一的。通过使用CSRF令牌，应用程序能够验证请求的合法性，防止攻击者利用受信任用户的身份进行未经授权的操作。

CSRF token能抵御XSS攻击吗？

假设服务器正确验证CSRF令牌并拒绝没有有效令牌的请求，那么令牌确实可以防止利用XSS漏洞。这里的关键在于名称中的”跨站脚本”，至少在反射型XSS的形式中，涉及到跨站请求。通过防止攻击者伪造跨站请求，应用程序可以防止对XSS漏洞的轻松利用。

由于CSRF令牌需要与每个请求一起发送，并且攻击者无法获得受害者的有效令牌，因此攻击者无法构造包含正确令牌的跨站请求。这样，即使存在XSS漏洞，攻击者也无法通过利用该漏洞发起跨站请求，从而限制了XSS漏洞的利用能力。

因此，在服务器正确验证CSRF令牌并拒绝无效令牌的情况下，CSRF令牌的确可以防止对XSS漏洞的利用。通过综合采用CSRF令牌和其他XSS防御措施，可以增强应用程序的安全性，减少跨站脚本攻击的风险。

CSRF与XSS的区别

跨站脚本攻击（XSS）允许攻击者在受害用户的浏览器中执行任意的JavaScript代码。

跨站请求伪造（CSRF）允许攻击者引导受害用户执行他们不打算执行的操作

。XSS漏洞的后果通常比CSRF漏洞更严重：

  • CSRF通常仅适用于用户能够执行的某些操作的子集。许多应用程序通常会实施CSRF防御，但可能会忽视其中一两个操作的保护。相比之下，成功的XSS利用可以诱使用户执行任何操作，而不受漏洞出现的功能的限制。
  • CSRF可以被描述为“单向”漏洞，因为攻击者可以引导受害者发起HTTP请求，但他们无法检索该请求的响应。相比之下，XSS是“双向”的，攻击者注入的脚本可以发起任意请求，读取响应，并将数据泄漏到攻击者选择的外部域名。

总结来说，XSS漏洞的后果通常比CSRF漏洞更严重。XSS漏洞可以使攻击者执行任意操作并从中获取数据，而CSRF漏洞仅限于特定的受影响操作。因此，在开发和维护应用程序时，需要同时关注XSS和CSRF防御，并采取适当的安全措施来减轻这些漏洞的风险。

单向与双向是考试重点！！！****

详细解释单向，双向

  1. CSRF - “单向”漏洞：

CSRF攻击利用了目标网站对用户身份验证不足的问题，攻击者可以通过某种方式引导受害者在目标网站上执行非预期的操作。这些操作是由受害者的浏览器自动发送的，攻击者并不能直接获取到该请求的响应。攻击者通常构造恶意网页或链接，诱使受害者点击，从而触发了事先构造好的请求。

例如，假设受害者在一个社交媒体网站上已经登录，并且在同一个浏览器上打开了一个恶意网页。该恶意网页包含一个图片标签，其src属性指向一个修改用户密码的请求。浏览器会自动发送该请求，但受害者并不知情。攻击者无法获取到该请求的响应，因此无法直接读取或窃取数据。

  1. XSS - “双向”漏洞：

XSS攻击是通过在目标网站的页面中注入恶意脚本，然后在用户浏览器中执行该脚本来实现的。这使得攻击者可以与目标网站进行”双向”通信，包括发送请求、读取响应和窃取数据。

通过XSS攻击，攻击者可以在受影响的网页中注入恶意脚本，并使其在用户的浏览器中执行。该脚本可以发起任意的HTTP请求，包括与目标网站的通信。攻击者可以通过这些请求读取目标网站的响应，并将获取到的数据发送到攻击者指定的外部域名。这使得攻击者能够窃取用户的敏感信息、劫持会话等。

需要注意的是，XSS攻击的成功注入和执行恶意脚本通常需要滥用目标网站上的输入字段、评论区或其他用户可输入的地方。通过注入恶意脚本，攻击者可以控制用户浏览器中的页面内容和行为，以达到自己的目的。

第六章 网络加密与密钥管理

密钥分配

什么是双钥体制

使用两个密钥，分别为公钥和私钥。

双钥体制的特点

  • 运算量大，不适合实时数据的加解密
  • 适合来进行密钥的分配
  • 采用Diffie-Hellman或RSA、ECC等公钥体制实现
  • 采用可信密钥管理中心（KDC）进行密钥分配

密钥分配的三种基本模式

  • 模式1:点-点密钥分配
  • 模式2:密钥分配中心（KDC）
  • 模式3:密钥传递中心（KTC）

点-点

A与B之间有共享的基本密钥kp

通过kp来传递会话密钥k

KDC（会话密钥k由KDC产生）

首先，A与KDC，B与KDC之间都有共享的基本密钥

A以明文向KDC发起会话密钥请求包（包括通信双方A，B的身份以及传输的唯一标识N1[临时交换号，可选时间戳，随机数或者计数器，KDC可根据临时交换号设计防重放机制]）

KDC返回的信息包括两部分

  • 第一部分是A想获取的信息，用A的主密钥KA加密，包括会话密钥Ks和KDC收到的请求包内容用以验证之前的消息是否被修改/重放
  • 第二部分是B想获取的信息，用A的主密钥KB加密，包括会话密钥Ks和A的身份，A收到这部分消息便原样发给B
  • [或者由KDC直接发给B，而不经过A]

为保证A发给B的会话密钥信息未被重放攻击，A B使用会话密钥进行最后的验证

B使用新的会话密钥Ks加密临时交换号N2并发送给A。A对N2进行函数变换后，用会话密钥发给B验证

KTC

KTC与KDC十分相似，只是会话密钥由A产生，而不是KDC

密钥分配协议

单钥体制

假设Alice和Bob都与KDC（Trent）共享一个密钥

协议描述：

  1. Alice呼叫Trent，请求得到与Bob通信的会话密钥
  2. Trent生成一个随机会话密钥，并做两次不同的加密（分别用与A和与B的共享密钥），Trent将两次加密的结果都发给Alice
  3. Alice采用共享密钥对属于她的进行解密得到会话密钥
  4. Alice将属于Bob的密文发送给Bob（也有说KDC直接发Bob）
  5. Bob解密后得到会话密钥
  6. Alice和Bob采用该会话密钥进行安全通信

双钥体制

【类似于SSL/TLS协议的会话密钥交换】

传统双钥体制的密钥交换容易遭受中间人攻击

协议描述：

  1. Alice收到Bob的公钥
  2. Alice生成一个随机的会话密钥k，采用Bob的公钥加密后，发送给Bob
  3. Bob采用其私钥对Alice的消息进行解密，得到随机的会话密钥k
  4. Bob和Alice采用会话密钥k对通信进行加密
怎么发起中间人攻击

中间人攻击是一个（缺乏）相互认证的攻击。大多数的加密协议都专门加入了一些特殊的认证方法以组织中间人攻击。例如SSL协议验证参与通信一方/双方使用的证书是否由权威的数字证书认证机构颁发（CA）来证明

Attacker怎么发起中间人攻击？

  1. Alice将她的公钥传给Bob的时候，Attacker截取了这个密钥，并改为传自己的公钥给Bob
  2. Bob将他的公钥传给Alice的时候，Attacker截取了这个密钥，并改为传自己的公钥给Alice
  3. 此时Alice和Bob手里拿的都是Attacker的公钥
  4. 当Alice用“Bob”的公钥加密k传给Bob时，被Attacker截取后用私钥解密，改用之前截获的Bob的公钥加密后，传给Bob
  5. 当Bob用“Alice”的公钥加密信息传给Alice时，被Attacker截取后用私钥解密，改用之前截获的Alice的公钥加密后，传给Alice

联锁协议（会考试）

  1. Alice发送公钥给Bob
  2. Bob发送公钥给Alice
  3. Alice用Bob的公钥加密后，将一半密文发送给Bob
  4. Alice用Alice的公钥加密后，将一半密文发送给Alice
  5. Alice发送另一半密文给Bob
  6. Bob将Alice发送的两部分组合后，用私钥解密
  7. Bob发送另一半密文给Alice
  8. Alice组合后，用私钥解密

为什么可以阻止中间人攻击？

  1. 在前两部分，Alice和Bob都拿到了假公钥（attacker的公钥）因为在步骤3，中间人无法恢复消息后再用Bob的公钥加密（因为Alice只发了密文的一半，只有拿到完整密文才能解密），所以只能虚构一个消息发送一半给Bob,...对话就完全不同了
  2. 如果在步骤3 直接转发Alice消息给Bob，那么当Bob收齐后，发现无法解密，那么就不会把自己另一半密文发出去
  3. 最常见的：中间人在第五步收齐Alice的之后，解密，根据语义伪造新消息，发送bob，但这样Bob只会以为这是一半的消息会一直等，总之会造成语义上的混乱
  4. 中间人不知道双方使用了传一半的方式，失败后直接放弃攻击

第三方CA认证（源头解决问题）

在会话密钥交换协议中采用第三方CA认证，即CA私钥数字签名技术（这也是SSL协议中会话密钥建立时使用的方法，是HTTPS抵抗中间人攻击的方法）

  • Trent 是一个可信的实体，对Alice和Bob的公钥都进行了数字签名
  • 签名后的公钥包含在一个数字证书中
  • 当Alice和Bob收到对方的公钥时，可以通过验证Trent的签名来确定合法性

数字签名：Trent用自己的私钥对信息进行摘要（hash计算，或叫指纹），再进行加密得到的Trent的数字签名，并将其附在数字信息上

假设Alice的公钥Kap，Trent对其进行hash计算得到摘要（指纹）H(Kap)然后使用Trent的私钥Kts加密后

Kts(H(Kap)) 即为数字签名

验证阶段：使用内置在系统里的CA证书中的Trent公钥Ktp解密后得到H(Kap)。由于Alice传输时是

Kap | Kts(H(Kap)) 的形式传输的，因此对Kap进行Hash运算后，与解密得到的摘要（指纹）进行比对，就可以验证

Alice发给Bob的公钥是经过CA认证签名的，Bob收到后可以验证

CA和KDC的区别

KDC是一个用于身份验证的中心，而CA是负责颁发和管理数字证书的机构。它们在网络安全中扮演不同的角色，但在某些情况下可能会有一定的关联
